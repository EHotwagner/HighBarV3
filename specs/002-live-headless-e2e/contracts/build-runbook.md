# Contract: `BUILD.md` literate-runbook format

**Addresses**: FR-022, SC-008.

`BUILD.md` at the repo root is a machine-validated, human-readable
runbook. It takes a maintainer from a clean checkout to a running
`spring-headless` match with an F# observer attached, in 10 or fewer
discrete steps. Every step is also a CI-executable assertion.

## Structure

- Markdown H2 headings of the form `## N. Step title` where `N` is
  a positive integer from 1 upward, contiguous.
- At most 10 such headings. Additional H2 headings (e.g.
  `## Prerequisites`, `## Troubleshooting`) are permitted but do not
  count toward the step count and do not need expect-comments.
- The "step" H2 headings are identified by a strict regex:
  `^## [0-9]+\. ` at the start of a line.

## Expect-comment convention

Each numbered step contains exactly one fenced code block tagged
`bash`. Immediately above the opening fence (no blank line between)
is an HTML comment of the form:

```
<!-- expect: <substring> -->
```

Where `<substring>` is a literal substring that must appear in the
step's combined stdout+stderr when the block is run. Substring
matching is case-sensitive, whitespace-sensitive, and ignores ANSI
color codes (the validator strips them before matching).

Only *one* expect-substring per step — if the step inherently needs
multiple assertions, compose them into a single distinctive line
(e.g., via a final `echo RUNBOOK-STEP-N-OK` in the bash block).

## Driver

The validator lives at
`tests/headless/build-runbook-validation.sh`. It:

1. Parses `BUILD.md` using a minimal markdown walker (no GitHub-flavored-md
   extensions required).
2. For each numbered step, extracts the fenced bash block and the
   expect comment.
3. Executes each block in order in a single bash subshell (shared
   working directory and environment across steps; steps may depend
   on each other).
4. On the first block whose combined stdout+stderr does not contain
   its `expect` substring, the validator exits non-zero with the
   step number, the expected substring, and the last 200 lines of
   output.

The validator runs on a GitHub-hosted `ubuntu-22.04` runner — that
is the "clean VM image" the spec's SC-008 names. Because each
GitHub-hosted job gets an ephemeral VM, no separate provisioning is
needed.

## Required step coverage

The 10 steps must end in a state where a running `spring-headless`
process is attached to an F# observer. Exact phrasing is up to the
maintainer, but the step *topics* are:

1. Pre-flight (OS, pinned `spring-headless`, vcpkg present).
2. Dependency hydration (vcpkg bootstrap or manifest install).
3. Proto codegen (`buf generate`).
4. CMake configure.
5. CMake build (plugin `.so` produced).
6. `ctest` (unit tests green).
7. F# client restore + build.
8. Python client codegen + editable install.
9. Launch engine (`spring-headless` starts, plugin loads, startup
   log line appears).
10. Attach observer (F# sample runs, first `Snapshot` logged).

Fewer steps acceptable; more than 10 is a hard failure.

## Relationship to `specs/001-grpc-gateway/quickstart.md`

Where the two documents overlap, `BUILD.md` is the source of truth.
The 001 quickstart is edited in this feature so every command it
names is a command `BUILD.md` also uses. Any remaining 001 quickstart
content that goes beyond `BUILD.md`'s ten steps (e.g. transport-mode
switching demonstrations) stays in the quickstart and is not
duplicated.

## Peer-walkthrough sign-off

SC-008 also requires at least one peer walkthrough on a clean VM
before the feature is marked complete. The walkthrough is tracked
in `specs/002-live-headless-e2e/checklists/peer-walkthrough.md` (to
be generated by `/speckit.checklist`) and requires one sign-off line
from a non-author maintainer who has executed `BUILD.md` end-to-end.
